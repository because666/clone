# AetherWeave 服务器性能推荐与版本映射方案

本文根据当前 AetherWeave 代码结构、部署文件和数据体量整理，回应群聊中两个问题：

1. 服务器上的运行版本要和 repo 版本建立统一映射，避免“仓库改了但线上未生效”。
2. 新服务器需要给出配置推荐，并预留未来高并发和流量增长空间。

## 1. 项目运行特征

AetherWeave 是“前端静态大屏 + Flask 算法服务”的一体化 Web 应用：

- 前端：React 19、TypeScript、Vite、MapLibre GL、Deck.gl、ECharts，主要压力在浏览器端 WebGL 渲染和静态资源下载。
- 后端：Flask + Gunicorn，提供登录鉴权、任务调度、SSE 状态流、A* 航线规划、ROI 分析、数据分析接口。
- 数据：`frontend/public/data` 约 91 MB，`data` 约 72 MB；单城市 3D 建筑 GeoJSON 最大约 14.7 MB，轨迹 JSON 每城约 3 MB。
- 算法：A* 典型单次规划约 50-200 ms，属于 CPU 突发型请求；批量生成航线会明显吃 CPU，不建议在比赛展示高峰期频繁跑。
- 持久化：默认 SQLite，适合演示和中小规模使用；高并发写入或多实例部署时应切换 PostgreSQL。
- 实时连接：`/api/tasks/stream` 使用 SSE，长连接会占用 Gunicorn 线程；如果未来上千连接，建议改 WebSocket/消息队列或单独拆出推送服务。

当前 `docker-compose.yml` 写的是 2 核小机器资源限制，适合临时演示，不建议作为最终比赛服务器配置。

## 2. 配置推荐结论

### 推荐购买配置：4 核 8 GB

如果目标是比赛答辩、项目展示、团队成员和评委稳定访问，建议直接购买：

| 项目 | 推荐值 |
|---|---|
| CPU | 4 vCPU |
| 内存 | 8 GB |
| 系统盘 | 80 GB ESSD/SSD |
| 带宽 | 20 Mbps 起步，预算允许选 30-50 Mbps |
| 系统 | Ubuntu 22.04 LTS |
| 部署方式 | Docker Compose + Nginx 反向代理 |
| 数据库 | 演示可 SQLite；长期运行建议 PostgreSQL |

理由：项目首屏会拉取地图、3D 建筑、轨迹、能耗等资源；后端还需要加载 POI/禁飞区索引并执行 A*。2 核 2 GB 能跑，但构建、缓存、并发请求和偶发批量计算都会比较紧，容易在演示时出现 504 或卡顿。4 核 8 GB 是性价比最高的稳妥档。

### 预留高并发配置：8 核 16 GB

如果要公开给更多同学/评委同时访问，或未来准备做线上长期演示，建议一步到位：

| 项目 | 推荐值 |
|---|---|
| CPU | 8 vCPU |
| 内存 | 16 GB |
| 系统盘 | 100-200 GB ESSD/SSD |
| 带宽 | 50-100 Mbps，或接 CDN |
| 数据库 | PostgreSQL/RDS，至少 2 核 4 GB |
| 静态资源 | Nginx 强缓存；更进一步放 OSS/CDN |
| 后端 | Gunicorn 4-6 workers，每 worker 4 threads |

这一档能更好支撑 500+ 并发浏览、更多 SSE 长连接和后台计算请求。真正大流量时，瓶颈大概率会先出现在静态资源带宽和 SSE 连接数，而不是单纯 CPU。

### 最低可用配置：2 核 4 GB

仅用于内部演示或备用环境：

| 项目 | 最低值 |
|---|---|
| CPU | 2 vCPU |
| 内存 | 4 GB |
| 系统盘 | 40 GB SSD |
| 带宽 | 10 Mbps |

不建议继续使用 2 核 2 GB 作为正式展示服务器。它可以跑起来，但留给 Python、Node 构建缓存、GeoJSON、Gunicorn 线程和系统本身的余量太小。

## 3. 并发容量估算

| 场景 | 服务器配置 | 预计承载 |
|---|---|---|
| 内部测试 | 2C4G + 10 Mbps | 20-50 人同时看大屏，少量任务操作 |
| 比赛答辩推荐 | 4C8G + 20-50 Mbps | 100-300 人浏览，20-50 个活跃操作用户 |
| 高并发预留 | 8C16G + 50-100 Mbps/CDN | 500-1000 人浏览，更多任务操作和 SSE 连接 |

估算前提：前端静态资源开启 gzip/brotli，浏览器缓存命中正常；批量轨迹生成不在访问高峰运行；数据库写入量不高。

## 4. 部署参数建议

当前已经把 Gunicorn 启动脚本改为环境变量可调：

```bash
WEB_CONCURRENCY=2
GUNICORN_THREADS=4
GUNICORN_TIMEOUT=120
```

建议值：

| 服务器 | WEB_CONCURRENCY | GUNICORN_THREADS |
|---|---:|---:|
| 2C4G | 2 | 4 |
| 4C8G | 3 | 4 |
| 8C16G | 4-6 | 4 |

如果 SSE 连接数增长明显，优先增加线程数或拆分推送服务；如果 A* 规划请求变多，优先增加 worker 数和 CPU。

## 5. 版本映射方案

为了解决“服务器版本和 repo 版本不一致”的问题，后端 `/api/health` 现在会返回构建信息：

```json
{
  "code": 0,
  "data": {
    "status": "healthy",
    "version": "1.0.0",
    "build": {
      "app_version": "1.0.0",
      "git_commit": "abcdef123456",
      "git_branch": "master",
      "build_time": "2026-05-06T12:00:00Z",
      "release_channel": "production"
    }
  }
}
```

部署时建议使用下面的流程：

```bash
export APP_VERSION=1.1.0
export VCS_REF=$(git rev-parse --short=12 HEAD)
export VCS_BRANCH=$(git rev-parse --abbrev-ref HEAD)
export BUILD_DATE=$(date -u +%Y-%m-%dT%H:%M:%SZ)

docker compose up -d --build
curl http://127.0.0.1:8080/api/health
```

上线后只要把 `/api/health` 里的 `git_commit` 和 GitHub/Gitee 仓库 commit 对上，就能确认线上运行的确实是这次代码。比赛演示前建议截图留档。

## 6. 进一步优化建议

- 不要用 `aetherweave:latest` 作为唯一线上标识，正式部署时可以额外打 `aetherweave:<commit>` 标签。
- Nginx 前置反代，开启 gzip/brotli、静态资源强缓存和 HTTPS。
- 静态数据目录可放 OSS/CDN，后端只承担 API 和算法请求。
- 批量生成航线、能耗预测等离线任务不要放在展示高峰执行，最好提前生成。
- 长期运行或多人同时改任务状态时，把 `DATABASE_URL` 切到 PostgreSQL。
- 若未来并发继续扩大，引入 Redis/Celery 做任务队列，将 A* 批量计算从 Web worker 中剥离。
