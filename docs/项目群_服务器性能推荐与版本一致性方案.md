# AetherWeave 服务器配置推荐与版本一致性方案

> 这份说明用于解决当前项目部署中的两个问题：  
> 1. 服务器运行版本和 repo 代码版本不完全一致，导致“仓库改了但线上不一定生效”。  
> 2. 需要根据 AetherWeave 当前项目规模，给出适合比赛演示和后续高并发预留的服务器配置建议。

---

## 一、先给结论

### 1. 推荐正式比赛展示服务器

建议购买或更换为：

| 项目 | 推荐配置 |
|---|---|
| CPU | 4 核 vCPU |
| 内存 | 8 GB |
| 系统盘 | 80 GB SSD / ESSD |
| 公网带宽 | 20 Mbps 起步，预算允许建议 30-50 Mbps |
| 操作系统 | Ubuntu 22.04 LTS |
| 部署方式 | Docker Compose + Nginx 反向代理 |
| 数据库 | 演示阶段 SQLite 可用；长期运行建议 PostgreSQL |

这个配置比较适合答辩、演示、评委访问、团队成员同时调试等场景。  
它比 2 核 2G 稳定很多，也不会像 8 核 16G 那样成本明显上升，是目前性价比最高的推荐档。

### 2. 如果要预留未来高并发

如果后续考虑公开访问、更多同学同时打开、或者长期挂在公网演示，建议直接上：

| 项目 | 高并发预留配置 |
|---|---|
| CPU | 8 核 vCPU |
| 内存 | 16 GB |
| 系统盘 | 100-200 GB SSD / ESSD |
| 公网带宽 | 50-100 Mbps，或配合 OSS/CDN |
| 数据库 | PostgreSQL / 云数据库 RDS |
| 静态资源 | 建议后续放 OSS/CDN |
| 后端并发 | Gunicorn 4-6 workers，每个 worker 4 threads |

这个配置可以更好支撑大量用户同时访问 3D 大屏、分析页、移动端 H5 页面，以及较多的任务创建/审批请求。

### 3. 不建议继续用 2 核 2G 做正式展示

2 核 2G 可以跑起来，但更适合临时 demo 或备用环境。  
原因是 AetherWeave 不是普通静态网页，它包含：

- React + Deck.gl + MapLibre 的 3D 大屏；
- 多城市建筑、POI、轨迹、能耗数据；
- Flask 后端 API；
- A* 航线规划；
- 任务调度与 SSE 实时状态流；
- 登录鉴权、审计、分析接口；
- Docker 构建和运行时内存开销。

2 核 2G 在演示时容易出现下面的问题：

- 首屏资源加载慢；
- 后端算法请求偶发超时；
- Docker 构建或重启时内存紧张；
- 多人同时访问时响应变慢；
- 服务器和 repo 版本不一致时排查成本高。

## 二、为什么 AetherWeave 对服务器有一定要求

当前 AetherWeave 是一个“前端 3D 可视化 + 后端算法服务”的完整 Web 系统，不是简单网页。

### 1. 前端资源比较重

前端使用：

- React 19；
- TypeScript；
- Vite；
- Deck.gl；
- MapLibre GL；
- ECharts；
- Web Worker；
- TypedArray / 二进制轨迹解码。

项目中静态数据体量较大：

| 数据类型 | 当前规模 |
|---|---|
| `frontend/public/data` | 约 91 MB |
| `data` | 约 72 MB |
| 单城市 3D 建筑 GeoJSON | 最大约 14.7 MB |
| 单城市轨迹 JSON | 约 3 MB |

这些资源在用户首次访问、切换城市、打开分析页时都会产生带宽和服务器静态文件传输压力。

### 2. 后端有算法计算，不只是查接口

后端使用 Flask + Gunicorn，提供：

- 登录认证；
- 用户角色权限；
- 任务创建；
- 任务审批；
- SSE 实时状态推送；
- A* 航线规划；
- ROI 沙盘分析；
- 城市分析数据接口；
- 轨迹 JSON / 二进制接口。

其中 A* 航线规划是 CPU 突发型任务。  
当前项目文档和测试结果显示，典型单条规划大约在 50-200 ms 级别。这个指标本身不错，但多人同时创建任务时仍然会占用 CPU。

### 3. SSE 长连接会占用后端并发资源

项目中的 `/api/tasks/stream` 使用 SSE 做任务状态推送。  
SSE 的优点是简单稳定，但每个打开页面的用户都可能保持一个长连接。

在小服务器上，如果很多用户同时打开页面，SSE 会占用 Gunicorn 线程，导致普通 API 请求排队变慢。

所以后续高并发时建议：

- 增加 Gunicorn threads；
- 前置 Nginx；
- 必要时将 SSE 改为 WebSocket；
- 或把实时推送模块单独拆出来。

## 三、不同配置的适用场景

| 配置 | 适用场景 | 风险 |
|---|---|---|
| 2 核 2G | 临时 demo、单人调试 | 内存余量太小，不建议正式展示 |
| 2 核 4G | 内部测试、小范围访问 | 多人并发和构建时仍然偏紧 |
| 4 核 8G | 比赛答辩、正式展示、团队协作 | 推荐档，稳定性和成本比较平衡 |
| 8 核 16G | 公开演示、高并发预留、长期运行 | 成本更高，但扩展空间更足 |

综合考虑比赛展示稳定性、项目体量和预算，建议优先选 4 核 8G。  
如果预算允许，并且希望一次性解决后续流量增长问题，可以直接选 8 核 16G。

## 四、建议的部署架构

### 当前阶段推荐架构

```text
用户浏览器
   ↓
Nginx
   ↓
Docker Compose
   ↓
AetherWeave 容器
   ├── 前端 dist 静态资源
   ├── Flask API
   ├── Gunicorn
   ├── SQLite
   └── 本地 data / logs 挂载目录
```

这种架构适合比赛演示，部署简单，排查方便。

### 后续高并发架构

```text
用户浏览器
   ↓
CDN / OSS 静态资源加速
   ↓
Nginx
   ↓
后端 API 服务
   ↓
PostgreSQL / RDS

后台任务：
Celery / Redis / 独立算法 Worker
```

后续如果访问量变大，建议把建筑 GeoJSON、轨迹 JSON、模型文件等静态资源放到 OSS/CDN，服务器只负责 API 和算法请求。

## 五、版本一致性问题怎么解决

目前群里提到的问题是：

> 服务器版本和 repo 版本不完全一致。  
> 如果只在 repo 里面改，同步到服务器不一定有效。

这个判断是对的。  
单纯 `git pull` 或本地改代码并不能保证线上一定运行的是最新版本，因为 Docker 镜像可能没有重新构建，容器可能没有重启，或者服务器上跑的不是当前 commit。

### 1. 新增版本指纹

现在建议让后端 `/api/health` 返回构建版本信息：

```json
{
  "code": 0,
  "data": {
    "status": "healthy",
    "version": "1.0.0",
    "build": {
      "app_version": "1.0.0",
      "git_commit": "abcdef123456",
      "git_branch": "master",
      "build_time": "2026-05-06T12:00:00Z",
      "release_channel": "production"
    },
    "database": "connected",
    "cached_cities": [],
    "uptime_seconds": 123.4
  },
  "message": "ok"
}
```

以后只要访问：

```bash
curl http://服务器IP:8080/api/health
```

然后看里面的：

- `git_commit`
- `git_branch`
- `build_time`
- `app_version`

就能判断服务器运行的版本是否和 repo 对得上。

### 2. 推荐部署流程

服务器上部署时建议按下面流程：

```bash
cd /opt/aetherweave

git pull origin master

export APP_VERSION=1.1.0
export VCS_REF=$(git rev-parse --short=12 HEAD)
export VCS_BRANCH=$(git rev-parse --abbrev-ref HEAD)
export BUILD_DATE=$(date -u +%Y-%m-%dT%H:%M:%SZ)

docker compose down
docker compose up -d --build

curl http://127.0.0.1:8080/api/health
```

重点是：  
每次改完 repo 后，不只是 `git pull`，还要重新 `docker compose up -d --build`，否则线上容器可能还是旧镜像。

### 3. 群里沟通时建议固定一个口径

以后每次部署完成，可以在群里发：

```text
AetherWeave 已部署：
版本：v1.1.0
分支：master
Commit：abcdef123456
构建时间：2026-05-06 12:00 UTC
访问地址：http://服务器IP:8080
健康检查：http://服务器IP:8080/api/health
```

这样大家就能确认：

- repo 是哪一次提交；
- 服务器跑的是哪一次构建；
- 页面打不开时应该排查代码、镜像、容器还是服务器网络。

## 六、Gunicorn 参数建议

当前项目生产环境使用 Gunicorn 启动 Flask 服务。  
建议把 worker 和 thread 做成环境变量，方便不同服务器配置调整。

### 推荐参数

| 服务器配置 | `WEB_CONCURRENCY` | `GUNICORN_THREADS` | 说明 |
|---|---:|---:|---|
| 2 核 4G | 2 | 4 | 内部测试可用 |
| 4 核 8G | 3 | 4 | 比赛展示推荐 |
| 8 核 16G | 4-6 | 4 | 高并发预留 |

示例：

```bash
export WEB_CONCURRENCY=3
export GUNICORN_THREADS=4
export GUNICORN_TIMEOUT=120
docker compose up -d --build
```

说明：

- 如果 A* 规划请求变多，优先增加 CPU 和 worker 数；
- 如果 SSE 长连接变多，优先增加 threads 或拆分实时推送服务；
- 如果静态资源加载慢，优先提升带宽或使用 CDN；
- 如果任务写入变多，优先把 SQLite 换成 PostgreSQL。

## 七、上线前检查清单

正式演示前建议检查：

### 1. 代码版本

```bash
git status
git log -1 --oneline
curl http://127.0.0.1:8080/api/health
```

确认 `/api/health` 返回的 `git_commit` 和 `git log -1` 一致。

### 2. 容器状态

```bash
docker compose ps
docker compose logs --tail=100
```

确认容器状态为 running / healthy。

### 3. 端口和安全组

需要开放：

| 端口 | 用途 |
|---|---|
| 22 | SSH，建议只允许团队固定 IP |
| 80 | HTTP，如果使用 Nginx |
| 443 | HTTPS，如果配置证书 |
| 8080 | 当前应用端口，如果直接暴露 |

正式环境建议通过 Nginx 暴露 80/443，不要长期直接暴露 8080。

### 4. 默认账号

项目默认账号是：

```text
admin / admin123
```

正式公网部署建议修改默认密码，避免被别人直接登录。

### 5. 演示数据

提前确认：

- 深圳、北京、上海、广州、成都、重庆等城市数据是否都能加载；
- 3D 建筑是否显示；
- 轨迹是否显示；
- 任务创建和审批是否正常；
- 移动端 `/mobile` 是否正常；
- 分析页 `/analytics` 是否正常；
- ROI 沙盘是否正常。

## 八、风险与后续优化

### 1. SQLite 风险

SQLite 对演示够用，但不适合多人高频写入和多实例部署。  
如果后续任务量变大，建议切换：

```text
PostgreSQL / 云数据库 RDS
```

### 2. 静态资源带宽风险

项目里 GeoJSON、轨迹、模型文件比较大。  
如果很多人同时打开页面，瓶颈可能不是 CPU，而是带宽。

建议后续：

- Nginx 开 gzip/brotli；
- 浏览器强缓存；
- 大文件放 OSS/CDN；
- 首屏只加载当前城市，其他城市延迟加载。

### 3. SSE 高并发风险

SSE 简单好用，但大量长连接会占线程。  
如果并发持续增加，建议改造为：

- WebSocket；
- Redis Pub/Sub；
- 独立实时推送服务。

### 4. 批量生成任务不要在展示高峰运行

批量轨迹生成会吃 CPU。  
建议提前生成好各城市轨迹数据，展示期间只做单条任务创建和审批。

---

## 九、最终建议

综合当前 AetherWeave 的功能复杂度、数据体量、比赛演示稳定性和后续扩展需求，建议如下：

1. 正式比赛展示服务器选 `4 核 8 GB + 20-50 Mbps`。
2. 如果预算允许并且想预留高并发，选 `8 核 16 GB + 50-100 Mbps`。
3. 部署时必须建立版本映射，确保 `/api/health` 能返回 repo commit。
4. 每次线上更新必须重新构建 Docker 镜像并重启容器。
5. 长期运行建议把 SQLite 升级为 PostgreSQL，把大静态资源迁移到 OSS/CDN。

一句话总结：  
**4 核 8G 是比赛展示的稳妥选择，8 核 16G 是面向后续高并发的保险选择；同时必须用 commit + build time 做服务器版本映射，避免 repo 和线上版本不同步。**
