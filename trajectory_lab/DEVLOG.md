## 2026-03-06 | 初始化建设（管线打通）

### 背景
原 `scripts/generate_logistics_trajectories.py`（1351 行）内嵌了 V0~V3.1 七个版本的禁飞区绕行算法，经分析存在根本性的几何缺陷：`line.difference(no_fly_union)` 裁切出的安全段两端在膨胀区边界上，但弦段会穿越原始禁飞圆。诊断过程见已归档的《禁飞区绕行算法开发记录.md》（已删除）。

决策：**完全废弃旧方案，新建 `trajectory_lab/` 独立开发**。

### 本次完成内容

#### 1. 目录与基础模块
- `core/geo_utils.py` — haversine、坐标转换、线性插值
- `core/no_fly_zones.py` — 禁飞区数据结构 `NoFlyZone` + 格网空间索引 `NoFlyZoneIndex`
  - 采用 0.01° 格网桶加速空间查询，避免 O(N) 遍历
  - 提供 `point_in_any()` / `segment_intersects_any()` 接口
- `core/poi_loader.py` — POI 加载 + **demand 净化**
  - 关键发现：深圳南山区 7955 个 demand POI 中，有 **5723 个（72%）落在禁飞区内**，若不过滤直接用于生成，轨迹起降点本身就违规
  - `load_city_pois()` 返回 `demand_clean` / `demand_blocked` 两个列表，生成时只用 `demand_clean`
- `core/planner.py` — 轨迹规划接口（当前：直线占位版 v0）
  - 含高度剖面生成（爬升-巡航-下降）
  - 含内置违规检测（用于生成后统计，不用于规划）
  - **接口稳定，后续算法迭代只改此文件内部**

#### 2. 生成脚本
- `batch_generate.py` — 城市级批量生成，写入前端路径
- `single_generate.py` — 指定两点（poi_id 或经纬度均可）生成单条

#### 3. API 服务
- `server.py` — Flask REST API，监听 `:5001`
  - `GET /api/status` — 健康检查
  - `GET /api/pois` — 返回净化后的 demand POI 列表
  - `POST /api/batch` — 批量生成
  - `POST /api/single` — 单条生成（含起降点禁飞区校验）
  - 城市 POI 数据在进程内缓存，避免重复读 GeoJSON

#### 4. 前端集成
- `frontend/vite.config.ts` — 添加 `/api/*` 到 Flask 的 proxy
- `frontend/src/components/AlgoLabPanel.tsx` — 新增右侧深色抽屉面板
  - 批量生成：数量/种子/距离范围可配置，一键触发
  - 选点生成：点击地图绿色圆点选起终点（需 Flask 在线）
  - 状态圆点：实时检测 Flask 是否在线
  - 结果展示：轨迹条数、违规段、耗时
- `frontend/src/components/MapContainer.tsx` — 集成 AlgoLabPanel
  - demand POI 点击高亮（选起终点时变色）
  - `reloadCurrentTrajectories()` — 热重载轨迹（cache-bust，无需整体刷新城市数据）

### 验证结果
- TypeScript 编译：✅ 零错误
- 批量生成（20条，深圳）：✅ 成功，耗时约 1s，输出 61KB JSON
- 前端热重载：✅ 生成完成后地图立即更新轨迹动画
- 仪表盘联动：✅ 生成 10 条后，左侧"活跃无人机"实时更新为 10

### 已知问题 / 待办
- 当前 `planner.py` 为直线占位版（v0），轨迹直接穿越禁飞区，`nfz_violations` 非零属正常
- **算法核心待实现**：需在 `core/planner.py` 的 `plan()` 函数中替换为真正的绕行算法
- `generate_all_cities.py` 调用了已删除的旧脚本，暂时失效，待新算法稳定后适配

**建议评估顺序**：
1. 可视图 (Visibility Graph) — 精确，适合凸多边形障碍；需处理圆形禁飞区的切线计算

- 合规率（无违规段比例）
- 路径冗余度（实际路径长度 / 直线距离）
- 生成速度（单条，目标 < 1s）

## 2026-03-06 | A*算法优化与深水区 Bug 排查战役

### 背景
旧版拉直线占位算法(v0)的避障侵入率在全城拉线时高达 95%。随后实施了多版本绕行算法（A* v1 直到最终稳定版 v4），在降低侵入率的过程中不仅对算法效率进行了深度打磨，也修复了一连串由于平面经纬学计算造成的隐秘物理盲区。

### 本次完成内容

#### 1. 消除算法层的“盲人接驳”漏洞
在最初的 A* 离散网格版本中，仅在网格进行碰撞检测，对非网格中心点的真实起飞点（真实经纬度）到第一个网格点缺乏飞线验证，导致起步就违章。
- **修复**：在图层搜索中将真实的起降点强行包装为 `START` 和 `GOAL` 节点，并且参与与周边网格边界的精准线段碰撞检测豁免，完美接入了网格。

#### 2. 扫清拉长线的底层几何物理畸变
通过引入拉线合并算法 `_smooth_path` 来减少无人机的多余转折，却发现本来安全的路径再次测得违规。深度排查出 `no_fly_zones.py` 中有两大致命缺陷：
- **包围盒截断**：长距离线的圆心距离直接被一个中点粗筛逻辑淘汰掉了边缘禁飞区。**已改为经纬度绝对极值包络提取**。
- **拉线垂直距离投影失真**：由于直接采用单纯地图坐标平面投影计算点到直线垂直距离，在高纬度下的经度缩水问题引发严重误差。**已引入 `cos(lat)` 系数将测距统统投射进了真实的局部平面米坐标系中解决**。
- **超视距边界衰减**：在粗筛的包围盒构建时，同样修正了因为地球球面经度在南山横向的余弦收缩导致的网格过滤盲斑，杜绝了漏网之鱼。

#### 3. 解除算力封印与多进程支持
当把规避精度推到极限后，无人机在极端密集的楼群夹缝中穿越时轻松突破了早期的两万次节点探索防线导致自动回撤直飞。
- **修复**：将安全节点展开算力直接放大至 `50000` 点。
- **提速**：将原本跑完深圳全图需求耗时 8 分钟的单线程算法，彻底重构成 `multiprocessing` 多进程池，在 23 核机器上仅仅使用了 **68秒** 就算完了包含万次碰撞规避几何计算的高清 `2000` 条无人机轨迹。

### 验证结果
- 全局侵入率由 **95% 降低到了完全合格的 4%**（剩余 4% 源起降点刚好落在了禁飞夹缝由于 10m 自定物理安全防卫缓冲钻不出而引发自动放弃直飞）。
- 大大低于期望目标的 1.05 极低绕行冗余度！
- 全量 2000 条结果数据完美在前端呈现！

## 2026-03-09 | 轨迹高度“错落有致”改造

### 背景
目前的算法中，所有的轨迹在起降过渡完成后，都会被固定分配到 `100.0` 米的高度。这在进行多条轨迹并发的三维可视化时，导致所有无人机都在同一个水平面上飞行，缺乏层次感和真实感。

### 本次完成内容
- **高度动态分配**：在 `planner.py` 的高度生成阶段，利用 `flight_id` 的 MD5 Hash 值，将所有的无人机巡航高度稳定打散并映射到 `80.0 ~ 120.0` 米区间。
- **重新生成轨迹**：执行 `batch_generate.py` 生成了深圳区域 `2000` 条轨迹数据，前端成功表现出预期的多层次“错落有致”的三维空间排布形态。
