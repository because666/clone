# trajectory_lab — 飞行轨迹算法需求文档

> 版本: v0.1 | 日期: 2026-03-06  
> 状态: 🔧 开发中（管线已打通，核心算法待实现）

---

## 一、背景与定位

本项目（`Anomaly Attribution Engine`）需要生成大量物流无人机飞行轨迹数据，用于城市低空空域异常检测与分析。

`trajectory_lab/` 是一个**独立的飞行轨迹算法开发沙盒**，与主项目其他代码完全隔离。其产物（轨迹 JSON）通过统一接口输出，供主项目前端直接消费。

**为什么单独开辟 trajectory_lab？**  
原有绕行算法（`generate_logistics_trajectories.py`，V0~V3.1，Shapely difference 方案）存在根本性的几何缺陷（弦段穿越原始禁飞区），已完全废弃。新算法在此独立迭代，主项目不受开发过程干扰。

---

## 二、核心需求

### 需求 1：城市级批量生成

- 给定城市（当前聚焦：深圳南山区），随机生成 N 条飞行轨迹
- N 可由调用方指定（1 到数千条）
- 每条轨迹表示一次 POI-to-POI 物流派送：从一个 demand POI 出发，抵达另一个 demand POI
- 起降点需预先净化：**落在禁飞区内的 demand POI 不得作为起降点**（见需求 4）

### 需求 2：指定两点实时生成

- 给定任意两个合规 demand POI（或直接给经纬度坐标），生成一条轨迹
- 实时响应（目标 < 3 秒）
- 生成前自动校验起降点合规性，若在禁飞区内则拒绝并报错

### 需求 3：城市接口化

- 所有算法接口通过 `city` 参数接入任意城市
- POI 数据路径约定：`data/processed/{city}/poi_demand.geojson` 和 `poi_sensitive.geojson`
- 当前优先保证深圳南山区的覆盖；其他城市（北京、上海、广州等）数据已就位，切换即用

### 需求 4：demand POI 净化

**重要前置步骤**：在生成任何轨迹前，必须过滤掉本身就位于禁飞区内的 demand POI。

原因：部分商业 POI（如学校内便利店、医院内药房）本身就在禁飞缓冲圆内，若以这些点为起降点，轨迹起点就已违规，数据不真实。

实测结果（深圳南山区）：
- demand 总数：7955 个
- 净化后可用：**2232 个**（约 28%）
- 被禁飞区覆盖：5723 个（约 72%）

> [!NOTE]
> 72% 的过滤率表明当前禁飞半径设置较严格。视算法迭代需要，可在 `poi_loader.load_city_pois(buffer_m=...)` 中通过 buffer 参数微调。

### 需求 5：前端实时审校

- 每次生成完成后，轨迹 JSON 自动写入前端可读路径
- 前端无需修改，刷新即可看到新轨迹
- 前端侧新增算法调试面板（右侧抽屉），支持直接触发生成、查看结果统计

## 三、轨迹格式规范

输出 JSON 与前端 `TripsLayer` 格式完全兼容：

```json
{
  "timeRange": { "min": 0, "max": 1234.0 },
  "totalFlights": 50,
  "trajectories": [
    {
      "id": "shenzhen_0001",
      "path": [[lon, lat, alt_scaled], ...],
      "timestamps": [0.0, 5.0, 10.0, ...]
    }
  ]
}
```

| 字段 | 说明 |
|------|------|
| `path` | `[经度, 纬度, 高度×3]`，高度单位为米乘以3（前端缩放约定） |
| `timestamps` | 秒，从0起，步长约5秒 |
| `id` | `{city}_{序号}` 格式 |

## 四、禁飞区规则

与前端 ColumnLayer 展示保持一致：

| 敏感类型 | 禁飞半径 (米) |
|----------|--------------|
| hospital / school | 300 |
| clinic / kindergarten | 250 |
| college / university | 200 |
| police | 150 |
| 默认 | 200 |

## 五、算法要求

### 当前版本（v0，占位）
- 直线飞行，不做任何绕行
- 用途：验证管线连通性，不用于正式数据生成

### 目标（v1 及以后）
核心约束：
1. **合规性**：路径上任意线段不得穿越禁飞圆（含小 buffer）
2. **自然性**：路径应尽量短且平滑，不产生不必要的大幅绕行
3. **性能**：单条轨迹生成 < 1 秒（深圳数据规模下）

候选算法方向（由开发者决定具体路线）：
- 可视图（Visibility Graph）+ Dijkstra/A*
- RRT / RRT*（概率路径规划）
- 基于 Shapely 的切线外绕（修正旧方案的弦穿越问题）
- 其他

**接口约定**：算法实现在 `core/planner.py` 的 `plan()` 函数中，签名不变，外部调用者无感知。
